# Требования к внешним интеграциям для сервисов «Умный дом»

## 1. Назначение интеграции

PropDevelopment планирует интеграцию с внешней партнёрской платформой «Умный дом». Платформа предоставляет два новых сервиса:

- интеллектуальный домофон;
- интеллектуальный шлагбаум.

Новые функции должны быть доступны собственникам через текущее мобильное приложение PropDevelopment.

Для интеграции предлагается использовать отдельный backend-сервис `smart-home-app`, который будет отвечать за управление функциями «Умного дома», взаимодействие с мобильным приложением, хранение минимального набора данных и обмен с внешней платформой партнёра.

---

## 2. Данные, участвующие в интеграции

В рамках интеграции обрабатываются следующие категории данных:

| Категория данных | Примеры данных | Где хранятся | Уровень критичности |
|---|---|---|---|
| Данные собственника | Идентификатор собственника, идентификатор дома, идентификатор ЖК | Основные данные — в CRM / tenant-core; в `smart-home-db` — только необходимые идентификаторы | Конфиденциальные |
| Данные доступа | Разрешения на открытие домофона, разрешения на въезд через шлагбаум | `smart-home-db` и внешняя платформа партнёра | Конфиденциальные |
| Данные автомобилей | Номер автомобиля, связь автомобиля с собственником или квартирой | `smart-home-db`, при необходимости передаются партнёру | Конфиденциальные |
| События доступа | Факт открытия двери, факт открытия шлагбаума, дата и время события, статус операции | `smart-home-db`, внешняя платформа партнёра | Конфиденциальные |
| Токены доступа | Access token, refresh token, временные токены для видеопотока | В защищённом хранилище секретов / в памяти сервиса; не должны храниться в открытом виде | Секретные |
| Видеопоток домофона | Видео в режиме реального времени | Не хранится в PropDevelopment, передаётся от партнёрской платформы в мобильное приложение | Конфиденциальные / секретные при наличии биометрии |
| Биометрические данные | Биометрические шаблоны лиц для распознавания | Не должны храниться в системах PropDevelopment; обрабатываются на стороне партнёра | Секретные |

## 3. Требования к безопасности

### 3.1. Общие требования

| № | Требование |
|---|---|
| 1 | Внешняя интеграция должна проходить только через контролируемый backend-сервис `smart-home-app`. Мобильное приложение не должно напрямую выполнять управляющие операции во внешней платформе без проверки прав в системах PropDevelopment. |
| 2 | Все взаимодействия между системами должны выполняться только по защищённым каналам связи: HTTPS/TLS, WSS, WebRTC с шифрованием. |
| 3 | Доступ к функциям «Умного дома» должен предоставляться только авторизованным собственникам или пользователям, которым собственник явно выдал доступ. |
| 4 | Необходимо исключить возможность доступа одного собственника к устройствам, автомобилям, событиям и объектам недвижимости другого собственника. |
| 5 | Все действия по открытию домофона, открытию шлагбаума, изменению разрешений и привязке автомобилей должны журналироваться. |
| 6 | Токены, ключи API и другие секреты не должны храниться в исходном коде, логах или открытом виде в базе данных. |
| 7 | Для интеграции с партнёром должны использоваться технические учётные записи с минимально необходимыми правами. |
| 8 | Должны быть настроены ограничения частоты запросов, таймауты и обработка недоступности внешней платформы. |
| 9 | Должен быть предусмотрен механизм отзыва доступа: удаление автомобиля, отключение пользователя, отзыв токена, отключение сервиса для квартиры или ЖК. |

---

### 3.2. Требования к персональным и чувствительным данным

| № | Требование |
|---|---|
| 1 | Номера автомобилей должны рассматриваться как конфиденциальные данные, так как они связаны с конкретным собственником, квартирой или ЖК. |
| 2 | Биометрические данные не должны храниться в системах PropDevelopment, если распознавание лиц выполняет партнёрская платформа. |
| 3 | При передаче данных партнёру необходимо передавать только те атрибуты, которые нужны для выполнения конкретной операции. |
| 4 | Необходимо фиксировать согласие пользователя на использование функций, связанных с распознаванием лица и обработкой биометрии. |
| 5 | Видеопоток домофона не должен сохраняться в системах PropDevelopment без отдельного основания и политики хранения. |
| 6 | Логи не должны содержать токены, полные персональные данные, биометрические данные или полные номера документов. |
| 7 | Для аналитики и отчётности данные должны передаваться только в обезличенном или минимизированном виде. |

---

### 3.3. Требования к доступу и разграничению прав

| № | Требование |
|---|---|
| 1 | Собственник должен иметь доступ только к устройствам, которые относятся к его квартире, дому или ЖК. |
| 2 | Пользователь не должен иметь возможность открыть домофон или шлагбаум другого ЖК. |
| 3 | Управление разрешениями доступа должно проверяться на уровне объекта: квартира, дом, ЖК, устройство, автомобиль. |
| 4 | Доступ к административным операциям должен быть отделён от доступа обычного собственника. |
| 5 | Все изменения прав доступа должны сохраняться в журнале аудита. |
| 6 | При смене собственника или отключении услуги права доступа должны автоматически отзываться. |

## 4. Аутентификация и авторизация

### 4.1. Аутентификация пользователей

Для пользователей мобильного приложения используется существующий механизм аутентификации PropDevelopment.

Рекомендуемый вариант:

- OAuth 2.0 / OpenID Connect;
- JWT access token;
- refresh token для продления сессии;
- MFA для критичных операций, если это предусмотрено политиками компании.

Мобильное приложение передаёт access token в `smart-home-app`. Сервис проверяет токен, определяет пользователя и получает его права на конкретные объекты.

---

### 4.2. Авторизация пользователей

Авторизация должна строиться на проверке прав пользователя к конкретному объекту.

Проверяются:

- идентификатор пользователя;
- роль пользователя;
- связь пользователя с квартирой;
- связь квартиры с домом или ЖК;
- доступность конкретного устройства;
- наличие разрешения на выполнение операции.

### 4.3. Аутентификация между системами

Для взаимодействия `smart-home-app` и партнёрской платформы рекомендуется использовать:

- OAuth 2.0 Client Credentials Flow;
- mTLS для дополнительной проверки сторон;
- короткоживущие access token;
- регулярную ротацию client secret / сертификатов.

`smart-home-app` не должен передавать партнёру пользовательские refresh token или внутренние токены PropDevelopment. Для внешней интеграции должен использоваться отдельный технический токен.
